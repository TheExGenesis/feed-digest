## Problem Statement

Francisco (Xiq) and Ivan want a shareable agent skill that consumes personal feeds (Twitter, Substack, blogs — later podcasts/YouTube), produces a **daily digest** tuned by memory of what each user has read and engaged with, delivers it via email + Telegram, captures free-form feedback (text or transcribed voice in a 1:1 agent conversation), and supports an optional **human-curation pass** that publishes a stable-URL "link post" others can subscribe to.

The skill must be portable — installable by `git clone` into any Claude Code skills directory, no hard dependency on Hermes-specific code (`hermes_log.py`, the Hermes event log, or Hermes folder conventions). Ivan's install has to work too.

## What Makes This Cool

1. **Convergence detection over time.** Noticing when three different blogs over three weeks are circling the same idea, and saying *that* — not just summarizing each post. No shipping tool does this well.
2. **Integration over past digests + memory, not raw sources.** The integration corpus is already-curated signal. Avoids LLM slop that happens when you feed it raw Twitter.
3. **Paragraph-level grounding.** When a post extends a thread you were stuck on last week, the digest points at the specific paragraphs to read.
4. **Two-layer digest with human-curated publish.** Layer A is your private daily digest. Layer B is what you keep after your feedback pass — published as a stable-URL "link post for everyone," with your comments inline. Your followers get your agent-mediated taste, not a generic feed.
5. **Shared group chat as ambient signal.** A Telegram group where both (or all) users' digests arrive and get argued over. The argument IS the high-quality feedback signal no thumbs-up button can match.

## Constraints

- **Portable SKILL.md** — works in Claude Code (Claude host), Hermes, Ivan's setup, and future hosts. No Hermes-specific imports.
- **Python + SQLite + cron** — same stack Xiq is already running, but packaged standalone.
- **Zero-server dogfood** — runs on each user's own machine or their own Hetzner-class server. No SaaS.
- **First-run config** — `config.yaml` sets source lists, email address, Telegram bot token + chat IDs, memory folder path. No account creation.
- **Markdown-first** — digests, memories, feedback are all markdown. No proprietary formats.

## Premises (agreed)

**P1. Portable SKILL.md.** Standalone, cross-harness compatible. No Hermes-specific deps.

**P2. Auto-ingest dir is separate from prestige `sources/`.** Raw fetched content lives in `feeds/raw/<source-type>/<source-id>/<date>.md`. Persists by default (not deleted).

**Prestige `sources/` defined:** Xiq's existing curated library. Structure: `sources/<title-slug>/source.md` (+ optional `notes.md`), YAML frontmatter with `date` and `url`. This folder pre-exists and is owned by the user. The skill never writes there except through explicit promotion.

**Promotion mechanism:** When user feedback includes an explicit promote marker (e.g., "save this to sources" / a `+sources` tag / a Telegram `/promote <item-id>` shortcut), the skill copies the corresponding raw item from `feeds/raw/...` to `sources/<title-slug>/source.md` with correct frontmatter. Implemented in Build step 7 (feedback listener) — not a separate component.

**P3. Integration corpus = past 7 days of digests + user memory folder (configurable path) + skill-internal feedback dir.** Explicitly NOT raw sources — avoids slop. Past digests are already-curated signal.

**Three distinct folders, clearly separated:**
- **User memory folder** (configured path, arbitrary — e.g., `~/hermes/memory/` for Xiq, whatever Ivan points to). Read-only from the skill's POV. Provides "who is this user and what do they care about" context.
- **Skill feedback dir** (`<skill-install-dir>/feedback/`). Owned by the skill. Every digest feedback message appends here (`feedback/<digest-id>.md`). Digest composer reads this as the **self-improving signal** — this is where the skill learns from the user over time. Lives with the skill so it travels when the skill is re-cloned.
- **Feeds raw dir** (`<skill-install-dir>/feeds/raw/`). Already defined in P2. Where fetched content lands. NOT read by digest composer (corpus comes from prior digests + memory + feedback, not raw).

**No default mixing with user's broader memory.** The skill does not write to the user memory folder. Feedback about the skill is separate and local to the skill.

**P4. Digests have stable URLs.** Served over HTTP (dashboard or equivalent). Delivery: email AND Telegram, both. Users pick their reading surface. Stable URLs make digest-to-digest and digest-to-memory backlinks work.

**P5. No real-time alerts in MVP.** Daily + weekly + on-demand digest for arbitrary date range only.

**P6. Twitter = community archive live API only for MVP.** Scoped to accounts in the archive. Out-of-archive accounts deferred to a future "hack." Substacks via `substack-api` (GitHub, no keys, free). Blogs via RSS (feedbar or equivalent).

*Implementation note:* Community archive API docs at their current docs page — confirm rate limits, auth model, and which accounts are actually in-archive *before* scaffold. If Ivan's followed accounts aren't in-archive, onboarding is blocked — resolve as prerequisite to Next Steps 4.

**P7. Feedback routed BY the host agent, not a listener daemon.** The skill has no Telegram bot, no long-poll, no webhook. Instead:

- The user sends a message to their own existing 1:1 agent (Hermes for Xiq; Ivan's own agent for Ivan).
- The agent detects "this is feedback for the feed-digest skill" using instructions in the skill's SKILL.md.
- The agent writes the feedback to `<skill-dir>/feedback/<digest-id>.md` using its normal file-edit tools.
- The agent also owns *delivery*: reads the digest markdown file the skill produced, sends it via email / Telegram / wherever, using its own tooling.

This means the skill's own Python code is pure cron + CLI. No daemons. The host agent is the live, always-on interface. **Ivan's equivalent agent needs tool access to file edits, Telegram, and email** for parity. If Ivan doesn't have that, the skill's `deliver.py` is a fallback CLI that sends email directly (Telegram delivery still requires an agent or a user-owned bot).

**Digest-id defined:** `YYYY-MM-DD` (date, ISO format). Used as digest filename, feedback filename, and correlation key across delivery/feedback/publish. If multiple on-demand digests happen in one day, append `-N` (e.g. `2026-04-16-2`).

**Commands vs feedback:** No skill-owned commands. Everything is natural language the agent interprets. The agent decides when to invoke `/promote`, `/publish`, etc. by calling the skill's CLI (`hermes-feed promote <item-id>`, `hermes-feed publish <digest-id>`). User speaks in plain English; agent chooses the tool.

**P8. Distribution = `git clone`.** Into `~/.claude/skills/` or the host's skill directory. First run writes `config.yaml` stub. Upgrade = `git pull`.

**P9. Two-layer digest.**
- **Layer A (private):** agent-generated raw digest → email + 1:1 Telegram personal chat. Primary reading surface.
- **Layer B (publishable):** after user's feedback pass, the agent **asks "can I publish?"** before producing a curated version with user comments inline. Published as stable URL. Optionally CC'd to followers via email or posted to the shared group chat.

**P10. Multi-player design, single-player MVP.** The config schema supports multiple users, but MVP runs as each user's own instance. Shared group-chat posting is the next milestone after single-player works. "One group for everyone" is fine for a long time.

**P11. Self-improving skill.** Feedback lives inside the skill install dir (`feedback/`), not in the user's memory folder. Digest composer reads its own accumulated feedback. Re-cloning the skill on a new machine loses local feedback unless the user commits it — intentional: feedback is personal state, not distributed code. Future: `./install --restore-feedback` could sync from a user-chosen location.

## Approaches Considered

### Approach A: Thin Skill (CHOSEN)
Ship end-to-end ingest→digest→deliver→feedback→publish on Python + SQLite + cron. Multi-user schema present; only single-user execution path in MVP.
- Effort: M (human 5-8 days / CC ~2-3 hrs) | Risk: Low-Medium | Completeness: 8/10
- *Revised upward from 3-5 days after spec review flagged Telegram bot wiring, prompt iteration, and two-machine smoke-testing as underestimated tails.*
- Pro: Fastest path to first digest. Every layer proven before adding multiplayer.
- Con: Group-chat social loop deferred to milestone 2.

### Approach B: Social Artifact
Build multi-user group-chat routing and publishable artifacts first. The digest is a shared object from day one.
- Effort: L (human 8-12 days / CC ~3-4 hrs) | Risk: Medium | Completeness: 9/10
- Pro: Social loop shipped up front. Ivan first-class from day one.
- Con: Multi-user wiring before first digest = risk of spending days on infra before discovering the digest is bad.

### Approach C: Corpus-First Experiment
Skip ingestion/delivery. One Python script produces tomorrow's digest from today's corpus. Iterate on prompts until output is good.
- Effort: S (human 2-3 days / CC ~60 min) | Risk: Low | Completeness: 5/10
- Pro: Forces digest-quality question first.
- Con: **Rejected — user doesn't already have a corpus of past digests, so this can't start cold.** Also skips the social loop which is where the reward lives.

## Recommended Approach — A, designed so B is easy

**Build order:**

1. **Skill scaffold.** `SKILL.md` + `pyproject.toml` + `config.example.yaml` (real `config.yaml` is `.gitignore`'d) + `scripts/` (ingestion/digest/deliver modules) + `feedback/` (self-improving state, gitignored by default) + `prompts/` (editable prompt files, committed). Standalone package, no Hermes imports. Local SQLite at `.feed/state.db` tracks seen items + digest history. Scripts like `substack-sync.py` are **copied** (not imported) from existing projects into this skill folder for modularity.

**SKILL.md = full operating manual for the host agent.** Sections required:

- **Setup (first run):** "Check that `config.yaml` exists; if not, copy `config.example.yaml` and prompt the user to fill in email/memory path/source lists."
- **Daily flow:** "If a new file exists in `digests/` that you haven't delivered (check `state.db`), read it, format it for the user's preferred surface, send it, mark delivered."
- **Feedback detection:** heuristics for identifying feed-related messages in 1:1 chat. "If the user's message references a recent digest, an article you sent, or uses phrases like 'more like this', 'less of X', 'save this one', it's feedback." Include 3-5 concrete examples the agent can pattern-match.
- **Feedback writing:** "Append feedback to `feedback/<digest-id>.md` with ISO timestamp and the user's verbatim message. Infer optional structured tags (`promote: <item-id>`, `don't_feature: <source-id>`, `tone: <positive|negative|neutral>`) and include them as YAML frontmatter — but never discard the verbatim message, since the prompt-iteration loop relies on it."
- **Item-id lookup:** "When the user references an item by name or topic (not by ID), grep the latest digest's markdown for matching text and resolve to the `[item-id]` shown inline."
- **Command invocation:** explicit mapping — "if user says 'save this', run `hermes-feed promote <item-id>`. If user says 'publish today's digest', run `hermes-feed publish <digest-id>`. If user says 'rerun digest for last week', run `hermes-feed digest --from <date> --to <date>`."
- **Publish gate:** "If latest digest has ≥1 feedback message and has not been published or skipped, prompt the user: *'I have your feedback on <digest-id>. Want me to publish a curated version?'* On yes/no/skip, take the corresponding action. If user ignores the prompt 3 days running, stop asking and mark auto-skipped."
- **Republish:** "If the user asks to republish after late feedback arrived, run `hermes-feed republish <digest-id>`."
- **Error recovery:** "If any CLI call fails (non-zero exit), tell the user exactly what failed and link to `logs/pipeline.jsonl`. Do not silently retry more than once."
- **Host-agent-agnostic:** written so any agent with file-edit + Telegram/email tools can follow. No Hermes-specific hooks.
   - **Secrets:** SMTP password, Telegram bot token, Claude API key live in `config.yaml` (gitignored) with env-var override (`FEED_TELEGRAM_TOKEN`, etc.). Never committed. `./install` checks for `config.yaml`; if missing, copies `config.example.yaml` and prompts user to fill it in.
   - **Logging:** `logs/pipeline.jsonl` append-only. Every ingest/digest/deliver/publish run writes a structured line with `ts`, `stage`, `status`, `error`. Rotated weekly.
   - **Migrations:** SQLite schema versioned via a simple `migrations/NNN_*.sql` folder + `state.db.schema_version` table. `./install` runs pending migrations idempotently. No alembic — overkill for MVP.
   - **Retry/failure:** ingest/deliver each wrap in try/except; on failure, log + retry 3x with backoff; after 3rd failure, send a Telegram DM to user chat ID saying "Ingest failed for <source>" with a link to `logs/pipeline.jsonl`.
2. **Ingestion layer.** One Python module per source type:
   - `sources/substack.py` — uses `substack-api` (GitHub repo, free, no keys)
   - `sources/twitter.py` — community archive live API, scoped to configured accounts
   - `sources/blog.py` — generic RSS via `feedparser` (or feedbar service for problematic feeds)
   - Each writes to `feeds/raw/<type>/<source-id>/<YYYY-MM-DD>.md` with YAML frontmatter (date, source, title, url). Idempotent — safe to re-run.
3. **Digest composer.** `digest.py` takes a date range:
   - Reads `feeds/raw/**/*.md` for the range (new since last digest)
   - Reads `digests/*.md` for past 7 days
   - Reads user memory folder (path from config `memory_path`, optional `memory_include_globs`)
   - Reads `<skill-dir>/feedback/*.md` — the skill's accumulated self-improving signal
   - Produces `digests/<YYYY-MM-DD>.md` + empty `published/` slot
   - Output spec: top items (3-7), convergence callouts, paragraph-pointers for thread continuations, prior-digest backlinks, source citations
   - **Token budget & truncation strategy — flexible, agent-driven, learns over time:**
     - Defaults: estimate total input token count from file sizes before the call. Budget = target Claude Sonnet context window minus output headroom (start at ~150K input / ~20K output).
     - **MVP — hardcoded solid defaults only:** single Claude Sonnet call; first-10K+last-5K truncation for any file >20K tokens with `[... <middle elided> ...]` marker; hard skip for any file >200K tokens; prune oldest raw items first if total corpus >150K tokens. No strategy flags, no sub-agents, no multi-pass pipeline.
     - **Deferred to milestone 2:** flexible token-budget orchestration (batch summarize via sub-agents, rank-and-prune by relevance, cheaper-model pre-filter). Revisit only if 2 weeks of real dogfooding proves single-pass is insufficient. Until then, hardcoded defaults stay.
     - **Learn from feedback:** when pruning drops items the user later asks about ("why wasn't X in the digest?"), log it to `feedback/` and update the prompt manually. No code change required for this loop.
4. **Delivery — host-agent primary, CLI fallback.** Two paths:
   - **Primary (agent-orchestrated):** cron runs `hermes-feed digest` → writes `digests/<YYYY-MM-DD>.md`. SKILL.md instructs the host agent: "every morning, check `digests/` for new files; read the latest one; send it to the user via whatever surface they prefer (email / 1:1 Telegram)." Host agent does the formatting + send using its own tool access.
   - **Fallback (`deliver.py` CLI):** for installs without an agent, or as a belt-and-braces. Sends email via SMTP (configured) only. Telegram is agent-only (we don't ship our own bot). Agent-mode can also call `hermes-feed deliver` explicitly if the user prefers that.
   - Both paths write the same stable URL (from step 5) into the message body.
5. **HTTP serving — static file host only (MVP).** Generate digests as plain HTML alongside markdown, serve via a simple process (`python -m http.server` or `caddy` one-liner) bound to a configured port. Unguessable slug in the URL provides enough obscurity for a personal-network artifact. Ship the same host for both installs — no Hermes-dashboard coupling. `use_existing_server` config flag is deferred. FastAPI path also deferred.
6. **Feedback routing via host agent (no listener).** SKILL.md includes a "How to handle user feedback" section read by the host agent. When the user sends a message in their 1:1 agent chat:
   - The agent checks: does the message reference a recent digest? (Correlate to the latest digest with `status='delivered'` in `state.db`, via `hermes-feed latest-digest-id`.)
   - If yes, the agent appends the message to `<skill-dir>/feedback/<digest-id>.md` with timestamp and any structured tags it infers (`promote: item-id`, `don't_feature: source-id`, `liked: reason`).
   - The agent invokes `hermes-feed promote <item-id>` if the user asked to save something to prestige sources.
   - The agent responds in chat confirming what it wrote.
   - **Item-id shown inline in each digest** (e.g., `[tw-abc12]`, `[sub-def34]`) so the user can reference specific items. The agent can look up item-id from free-form text ("the Zach Mandeville one") by grepping the latest digest.
   - **Self-improving loop:** next day's digest composer reads `feedback/*.md` and adapts. No code change; the prompt + learned examples do the work.
7. **Publish gate (Layer B) — host-agent orchestrated.** `publish.py` is a CLI the host agent invokes. Flow:
   - After a digest has been delivered and at least one feedback message captured, SKILL.md instructs the host agent to ask the user: *"I have your feedback on digest 2026-04-16. Want me to publish a curated version? (yes / no / skip)"*
   - On `yes`: agent runs `hermes-feed publish <digest-id>`. CLI reads latest digest + feedback/<digest-id>.md, produces `published/<YYYY-MM-DD>.md` (digest with un-starred items deleted, user comments inline), returns the stable URL. Agent posts the URL to configured output channels (email to followers, shared group chat).
   - On `no`: agent waits for more feedback, re-asks next morning.
   - On `skip`: no publish for this digest. Agent marks terminal in `state.db`.
   - **Stateless timeout:** no daemon, so no true timeout — the agent re-asks the next morning. If user ignores three consecutive asks, agent stops asking (logs "auto-skip after 3 ignored prompts" in `state.db`).
   - **Late feedback:** after publish, new feedback appends to `feedback/<digest-id>.md` with `post_publish: true`. Published artifact is NOT regenerated. Agent offers `hermes-feed republish <digest-id>` on user request.
   - **Minimum-viable Layer B:** copy the digest, delete un-starred items, inline user comments. No fancy re-summarization. ~0.5 day of work.
8. **Cron.** Daily at 07:00 local (configurable TZ via `zoneinfo`): `hermes-feed ingest && hermes-feed digest`. That's it. The host agent handles everything downstream — reading the new digest, sending it, waiting for user feedback, prompting publish. Cron is pure fetch-and-compose; the human/agent loop is async on top.

**Multi-player readiness (don't build, do design):**
- `config.yaml` has `users:` as a list. MVP uses `users[0]`. B milestone adds loop.
- `published/` URLs are public-by-default (unguessable slug, not auth'd).
- A future `group.py` will post all users' digest URLs into a configured Telegram group. Not in MVP.

## Open Questions

1. **Convergence detection quality.** Does "3 blogs circling the same idea over 3 weeks" actually emerge from the prompt, or do we get LLM word-soup? Only dogfooding 2-3 weeks of real digests will tell. *Mitigation:* keep the prompt in a single editable file `prompts/digest.md` so iteration is cheap.
2. **Memory folder structure.** User's memories are markdown but no schema yet. What counts as "engaged with"? Anything the user commented on? Anything in a specific subfolder? *Decision deferred:* start by reading `memories/**/*.md` ambient; refine after first week.
3. **Dashboard coupling.** Xiq has a Hermes dashboard on :7777 already; Ivan doesn't. Default to shipping own `serve.py`; have a config flag `use_existing_server: <url>` for hosts that want to reuse their own.
4. **Blog RSS via feedbar vs plain `feedparser`.** Feedbar is nicer for pesky feeds but adds a dependency. *Default:* `feedparser` first, feedbar only if specific feeds break.
5. **Timezone handling.** Cron fires "07:00 local" but "local" for a server is UTC. Skill config should take a TZ; cron fires in that TZ.
6. **Digest cardinality.** How many items per digest? 3? 10? *Start with:* top 5-7, with an "other notable" tail of up to 10 links.

## Success Criteria

**Dogfood schedule:** week 1 = Xiq solo (validates pipeline). Week 2 = Xiq + Ivan (validates portability and two-user flow). Success criteria measured across the full 2 weeks.

Measurable:
- **Reliability:** Xiq receives ≥6 of 7 daily digests in week 1 without pipeline failure. Week 2: Xiq and Ivan each receive ≥6 of 7. Failures logged to `logs/pipeline.jsonl` and surfaced via Telegram DM on repeated failure.
- **Integration works:** at least 3 digests per person contain a convergence callout that the user rates "non-obvious / useful" in feedback.
- **Feedback loop works:** at least 5 items per person get promoted to prestige `sources/` via feedback over the 2 weeks.
- **Publish loop works:** at least 3 Layer-B publishes happen per person. At least one gets referenced or linked by the other user in a Telegram message.
- **Operation cost:** under 30 minutes per week of operating the skill (updating config, restarting services, debugging). If it's more, the skill is too fragile.
- **Subjective signal:** at the end of week 2, each user can answer "would you pay \/month for this vs. Readwise Reader?" with a direct yes or no based on dogfood experience. Not a pricing decision — a is-it-better-than-the-alternative check.
