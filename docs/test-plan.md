**Framework:** pytest + monkeypatch + tmp_path (match hermes-workspace convention).

## Affected Pages/Routes

This skill has no web UI. Surfaces to test:
- CLI commands (`hermes-feed ingest`, `hermes-feed digest`, `hermes-feed publish`, `hermes-feed promote`, `hermes-feed deliver`, `hermes-feed digest --from --to --to-on-demand`)
- File system state (`digests/`, `published/`, `feedback/`, `feeds/raw/`, `sources/<slug>/`)
- SQLite state (`.feed/state.db` — tables: `sources`, `seen_items`, `digests`, `deliveries`, `publishes`, `schema_version`)
- Log files (`logs/pipeline.jsonl`)
- Host-agent integration (SKILL.md instructions — tested by reading the SKILL.md in isolation and asserting it covers the required sections)

## Key Interactions to Verify

**Ingest pipeline:**
- Substack source: given a mocked `substack_api.Newsletter` returning 3 posts (2 new, 1 seen), verify only new posts are written as markdown in `feeds/raw/substack/<url>/<date>.md`, and `state.db.seen_items` gets 2 new rows. Idempotent: second run with same input is a no-op.
- Twitter community archive: given a mocked HTTP response with 5 tweets for an in-archive account, verify 5 markdown files written. For an out-of-archive account, verify a warning log entry and clean exit.
- Blog RSS: given a feedparser fixture, verify N new posts written with correct frontmatter (date, url, title, source).
- Network timeout on any adapter: verify 3x retry with backoff, then log + skip that source, pipeline continues with other sources.

**Digest composer:**
- Happy path (small corpus): given fixtures of 3 raw substack posts + 2 past digests + a small memory folder, assert `digests/<date>.md` is created, contains item-ids referenced in raw feeds, and has basic structure (headings, links).
- Truncation: given a raw file of 50K tokens, verify the file is read as first-10K + `[... <middle elided> ...]` + last-5K before being sent to LLM.
- Hard skip: given a raw file of 250K tokens, verify it is excluded and logged.
- Corpus pruning: given corpus totaling 180K tokens, verify oldest raw items are pruned first until total ≤150K.
- Empty-day handling: given 0 new raw items in the range, verify digest is SKIPPED (no LLM call, no file write), and `logs/pipeline.jsonl` gets a `{"stage": "digest", "status": "skipped_no_new_items"}` entry.
- LLM retry: given a mock Anthropic client that returns 429 twice then success, verify the digest is eventually written.
- LLM hard failure: given 3 consecutive 429s, verify `digests/<date>.md` is NOT created, `failures/<date>.md` IS created with the last error, and `state.db.digests.status = 'failed'`.
- Malformed LLM response: given a mock returning garbage, save raw response to `failures/<date>.md` and log.

**Delivery:**
- `deliver.py --dry-run`: writes to `delivered/dryrun/<date>.eml` instead of sending. Assert file exists with correct subject, recipients, body (containing stable URL).
- SMTP auth fail: mocked SMTP raising `AuthenticationError` → logs, does not crash pipeline.
- Recipient unreachable (SMTPRecipientsRefused) → log specific recipient, continue.

**Promotion (`hermes-feed promote <item-id>`):**
- Happy path: given an item-id that exists in `feeds/raw/<type>/<src>/<date>.md`, verify copy to `sources/<title-slug>/source.md` with correct frontmatter (date, url, source). state.db records the promotion.
- Conflict — slug already exists: given `sources/post-title/source.md` exists, verify new item is written to `sources/post-title-2/source.md`. If `-2` exists, try `-3`, etc.
- Unknown item-id: exit 1, error to stderr identifying the missing id.

**Publish (Layer B):**
- Happy path: digest + ≥1 feedback entry → `published/<date>.md` written with un-starred items deleted and user comments inline. state.db `publishes.url` populated. Return URL on stdout.
- Already published: no-op, return existing URL on stdout.
- No feedback yet: exit with warning "no feedback on <date>, publish anyway? use --force".
- Republish after late feedback: `hermes-feed republish <date>` regenerates `published/<date>.md` with new feedback included.

**State DB migrations:**
- Fresh install: apply all numbered migrations in `migrations/`, set `schema_version` to latest.
- Existing install: only apply migrations with version > current. No-op if up-to-date.
- Failed migration: rollback transaction, `schema_version` unchanged, error raised.

**SKILL.md contract check (static):**
- A test reads `SKILL.md` and asserts it contains sections for: Setup, Daily flow, Feedback detection, Feedback writing, Item-id lookup, Command invocation, Publish gate, Republish, Error recovery. (Use regex or header-based checks — catches accidental deletion during refactor.)

## Edge Cases

- Multi-timezone: user in UTC+2 runs cron at "07:00 local" → verify `zoneinfo` resolves config TZ correctly.
- Unicode in source titles: Substack/blog titles with emoji, non-Latin scripts → verify slugification produces ASCII-safe folder names (use `python-slugify` or similar).
- Item-id collision across sources: Twitter `[tw-abc12]` and Substack `[sub-abc12]` — prefix-based IDs prevent this by construction. Test asserts IDs always have source prefix.
- Config file missing: `./install` offers to copy example; subsequent CLI calls without config fail with clear error.
- Stale lockfile: if digest.py crashes, `.feed/.lock` remains. Next run detects stale lock (age > 1h) and proceeds. Tested with a fixture-created lock file.

## Critical Paths (must work end-to-end)

1. **First-run setup E2E:** clone → `./install` → fill config.yaml → cron installed → first manual `hermes-feed ingest && hermes-feed digest` produces a digest file and a delivered email (dry-run). Asserts no errors, correct directory layout.
2. **Daily loop E2E:** fixture corpus → cron-equivalent run → digest file → mock host-agent delivery → simulated feedback write to `feedback/<id>.md` → next day's digest reads prior feedback. Asserts feedback influences the next digest (via a testable prompt assertion, e.g., "feedback section present in the prompt context").
3. **Publish E2E:** digest + feedback → `hermes-feed publish <id>` → `published/<id>.md` exists with correct content → URL returned. Asserts re-publish is a no-op.
4. **Failure recovery:** simulate Claude API outage for 1 day → pipeline logs skip, doesn't corrupt state.db → next day's run succeeds normally.

## EVAL (not unit tests)

**Golden-fixture prompt eval:**
- `tests/eval/fixtures/<scenario>/` contains: `raw/*.md` + `past-digests/*.md` + `memory/*.md` + `expected-digest.md`.
- `tests/eval/run_eval.py` runs `digest.py` against each fixture, compares output to expected. Use semantic diff (line-by-line + keyword match) rather than exact string match — LLM output varies.
- Scenarios to seed: (a) convergence detection (3 sources, same theme across 2 past digests), (b) paragraph pointer (memory mentions "stuck on X", new post discusses X), (c) empty day (no raw items), (d) integration with feedback (prior feedback said "more by author Y" → this digest should feature author Y prominently if present).
- Run on every prompt change (`git diff prompts/digest.md` → trigger eval).
- Baseline thresholds: digest contains ≥3 distinct items from raw corpus; includes at least one backlink to a past digest if one exists; includes at least one memory reference if relevant memory exists.
